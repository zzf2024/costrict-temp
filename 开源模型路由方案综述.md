# 开源模型路由方案综述

有，而且已经形成了几类不同方案。需要区分：

1. **能力路由**：判断问题难度，决定使用便宜模型还是贵模型。
2. **语义路由**：根据任务领域，把问题发送给代码、数学、翻译等专用模型。
3. **工程路由**：根据价格、延迟、限流和服务可用性选择部署节点。
4. **级联路由**：先调用便宜模型，结果不可靠时再升级到贵模型。

## 值得关注的开源项目

| 项目 | 路由类型 | 成熟度 | 适合场景 |
|---|---|---:|---|
| **RouteLLM** | 强弱模型能力路由 | 较成熟的研究框架 | 便宜模型与贵模型二选一 |
| **LiteLLM Router** | 网关、负载、价格、延迟、回退 | 生产工程较成熟 | 统一管理多个模型 API |
| **vLLM Semantic Router** | 信号、语义、策略、隐私和安全路由 | 新但功能完整 | 自托管、多模型生产系统 |
| **LLMRouter** | 多种学习型路由算法 | 研究功能丰富 | 训练和比较自己的路由器 |
| **Semantic Router** | Embedding/意图语义路由 | 轻量 | 按任务类别选择模型或 Agent |
| **cascade-routing** | 路由与级联结合 | 偏研究 | 先便宜后昂贵的质量级联 |
| **NVIDIA LLM Router** | 意图分类和神经网络路由 | 实验性 | 多模态和 NVIDIA 技术栈 |

## 1. RouteLLM：最接近“便宜/贵模型自动选择”

RouteLLM 由 LMSYS 团队开源，核心输入是：

- 一个弱模型，例如本地小模型或便宜 API；
- 一个强模型，例如高级推理模型；
- 一个路由阈值。

路由器预测强模型相对于弱模型的胜率：

\[
P(\text{strong model wins}\mid x)
\]

超过阈值就调用贵模型，否则调用便宜模型。

它提供：

- OpenAI 兼容服务；
- Matrix Factorization、BERT、加权 Elo 和 causal LLM 等路由器；
- 阈值校准；
- 训练、评测和自定义 Router 接口。

项目地址：

- https://github.com/lm-sys/routellm

需要注意：官方预训练路由器最初主要基于 `GPT-4-1106-preview` 与 `Mixtral-8x7B` 的偏好数据训练。虽然项目声称可以泛化到其他模型组合，但对于当前模型以及 CUDA 编程任务，最好使用自己的数据重新校准或训练。

适合的典型场景：

```text
简单代码解释、README、测试模板
    → 便宜模型

复杂 CUDA 调试、架构设计、数学推导
    → 贵模型
```

## 2. LiteLLM：最适合作为底层模型网关

LiteLLM 可以统一接入 OpenAI、Anthropic、Gemini、Bedrock、本地 vLLM、Ollama 等模型，并对外暴露统一接口。

它原生支持：

- weighted pick；
- rate-limit-aware；
- least-busy；
- latency-based；
- cost-based；
- fallback、retry、cooldown。

项目地址：

- https://github.com/BerriAI/litellm
- https://docs.litellm.ai/

需要注意：

> LiteLLM 默认解决的是“应该选择哪个部署节点”，不等于它能准确判断“这个问题需要聪明模型还是便宜模型”。

例如 cost-based routing 会选择价格更低的部署，但不一定理解 CUDA race condition 比 README 修改更困难。

因此推荐把 LiteLLM 放在底层：

```text
你的能力路由器
      ↓
LiteLLM Gateway
      ↓
OpenAI / Anthropic / 本地 vLLM / Ollama
```

## 3. vLLM Semantic Router：功能完整的生产方案

vLLM Semantic Router 面向 Mixture-of-Models 路由，可以综合：

- 关键词；
- 领域分类；
- Embedding 相似度；
- 上下文长度；
- 用户权限；
- 隐私约束；
- 成本和延迟；
- 安全与 jailbreak 检测；
- 本地、私有云和外部 API 边界。

它可以把多个信号组合成可配置决策，并支持 OpenAI 格式、多轮对话、多种模型提供商和 Kubernetes/Envoy 部署。

项目地址：

- https://github.com/vllm-project/semantic-router

适合：

- 同时使用本地模型与云端模型；
- 有多个项目或用户；
- 需要隐私、权限和安全策略；
- 未来准备把路由系统部署成公共服务。

对于个人 CUDA 项目，它可能略重；等路由规模扩大后再使用比较合理。

## 4. LLMRouter：适合研究和训练自己的路由器

UIUC 的 LLMRouter 集成了多种路由算法，包括：

- KNN、SVM、MLP；
- Matrix Factorization；
- Elo Router；
- RouterDC；
- Graph Router；
- causal LLM Router；
- 多轮 Router；
- 个性化 Router；
- Agentic Router。

它还提供训练、推理、数据生成、CLI 和可视化接口，并支持本地 OpenAI-compatible 模型服务器，如 Ollama、vLLM 和 SGLang。

项目地址：

- https://github.com/ulab-uiuc/LLMRouter

适合：

- 想研究不同 Router 算法；
- 想写论文或做实验；
- 想使用自己收集的代码问题数据训练路由器；
- 候选模型不止两个。

不太适合一上来直接当作简单生产网关。

## 5. Semantic Router：轻量级语义分类

Aurelio Labs 的 Semantic Router 使用 Embedding 空间判断语义类别，不需要每次再调用一个大模型做路由。它支持本地 Hugging Face Encoder 和本地 Llama.cpp。

项目地址：

- https://github.com/aurelio-labs/semantic-router

例如定义几个路由：

```text
documentation
python_basic
cuda_debugging
architecture
math_derivation
```

根据相似度选择不同模型：

```python
routes = {
    "documentation": "cheap-model",
    "python_basic": "cheap-model",
    "cuda_debugging": "strong-model",
    "architecture": "strong-model",
    "math_derivation": "strong-model",
}
```

优点：

- 快；
- 成本非常低；
- 规则容易解释；
- 不需要训练复杂的路由模型。

缺点是它主要识别“任务属于什么类别”，不一定能区分同一个类别内的简单问题和困难问题。

## 6. cascade-routing：先便宜模型，必要时再升级

ETH Zürich 的 `cascade-routing` 将模型路由和模型级联统一起来。

例如：

```text
便宜模型生成答案
      ↓
质量评估器判断是否足够好
      ↓
不够好则调用更强模型
      ↓
仍不够好则继续升级
```

项目地址：

- https://github.com/eth-sri/cascade-routing

项目要求用户定义自己的质量估计器和成本估计器，因此灵活，但不是安装后即可直接用于生产的完整应用。

它很适合代码开发，因为代码任务具有天然的质量信号：

- 是否编译通过；
- pytest 是否通过；
- 数值误差是否达标；
- sanitizer 是否报错；
- benchmark 是否退化。

相比仅分析用户 Prompt，这类执行反馈级联通常更可靠。

## 7. NVIDIA LLM Router：实验性多模态方案

NVIDIA 的实验性 Blueprint 提供：

- 基于小模型的意图分类；
- CLIP Embedding 加神经网络自动路由；
- 文本和图像多模态请求；
- OpenAI-compatible 接口；
- 自定义模型映射和重新训练。

项目地址：

- https://github.com/NVIDIA-AI-Blueprints/llm-router

官方将其定位为实验性 Blueprint，默认配置也比较依赖 NVIDIA 服务和容器技术栈。

比较适合多模态研究，不是当前代码开发路由的第一选择。

# 针对 CUDA/PyTorch 开源项目的推荐组合

建议不要只选择一个项目，而是组合：

```text
Codex、IDE 或开发 Agent
          ↓
自定义规则与语义分类
          ↓
RouteLLM 或简单评分器
          ↓
LiteLLM Gateway
       ↙        ↘
便宜模型        贵模型
          ↓
编译、测试、sanitizer、benchmark
          ↓
失败则自动升级贵模型
```

## 第一阶段：最小可用版本

使用：

- **LiteLLM**：统一模型 API；
- **Python 硬规则**：判断任务类型；
- **pytest/编译结果**：控制升级。

例如：

```python
STRONG_KEYWORDS = {
    "race condition",
    "illegal memory access",
    "shared memory",
    "warp",
    "数值稳定性",
    "架构设计",
    "公共 API",
}

def select_model(task: str) -> str:
    normalized = task.lower()

    if any(keyword in normalized for keyword in STRONG_KEYWORDS):
        return "strong-model"

    return "cheap-model"
```

## 第二阶段：加入 Semantic Router

用 Embedding 将任务分成：

- 文档；
- 简单 Python；
- 测试生成；
- CUDA 实现；
- CUDA 调试；
- 算法与数学；
- 架构审查。

## 第三阶段：积累项目数据

记录每次任务：

```json
{
  "task": "修复非2次幂shape下的RMSNorm错误",
  "selected_model": "cheap",
  "tests_passed": false,
  "escalated": true,
  "strong_model_succeeded": true,
  "cost": 0.08
}
```

积累几百条以后，再使用 RouteLLM、LLMRouter 或自己的分类器训练项目专用路由器。

# 最值得优先尝试的三个方案

## 最快落地

\[
oxed{\text{LiteLLM + 自定义规则 + 测试失败自动升级}}
\]

## 自动判断强弱模型

\[
oxed{\text{RouteLLM + LiteLLM}}
\]

## 未来做复杂、自托管生产系统

\[
oxed{\text{vLLM Semantic Router}}
\]

对于目前的开发阶段，最推荐第一种。因为代码任务具有编译器、测试和 sanitizer 这些客观反馈，不需要一开始就训练复杂路由模型。
