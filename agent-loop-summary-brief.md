# CoStrict Agent Loop 改进工作汇报

---

## 一、背景

CoStrict 的 agent loop 基于 Roo Code 二次开发，采用前端主导循环 + 后端 prompt 处理的架构。在实际使用和 SWE-bench 评测中，发现原始版本存在工具调用效率低、死循环、无界轮次等问题，影响用户体验和任务完成质量。本次工作针对这些问题进行了系统性改进，并通过 SWE-bench Lite 评测验证了效果。

---

## 二、原始版本存在的问题

### 问题 1：工具调用效率低，大量命令执行失败

**现象**：原始评测中命令执行成功率仅 51.5%，32 次命令执行失败。

**根因**：后端 prompt 处理链只做消息过滤和规则注入，不感知项目环境。Agent 收到的 system prompt 中没有任何关于项目测试运行方式的信息，只能凭模型自身知识猜测。而模型对 Django 项目的测试方式存在系统性误解——优先使用通用的 `python -m django test` 和 `python -c` 内联脚本，但这些命令在 Django 项目中因 settings 模块路径未配置而必然失败（`ImproperlyConfigured`、`ModuleNotFoundError`）。后端作为每轮 prompt 的必经处理层，本可以补充这一信息，但原始版本未利用这一位置优势。

### 问题 2：存在死循环风险

**现象**：10 个实例中 1 个触发死循环，最严重的实例耗时 766 秒、调用工具 45 次。

**根因**：两层缺失叠加。第一层，前端虽有 `ToolRepetitionDetector`（连续相同工具调用 ≥3 次拦截），但只能检测"完全相同"的重复——agent 每次微调命令参数（换个测试路径、加个环境变量）就能绕过检测，而本质上仍在用错误方式反复试错。第二层，后端对命令执行历史完全无感知，不追踪同一任务执行过哪些命令、哪些失败了，无法识别"换汤不换药"的试错模式。agent 在缺乏正确指引的情况下不断尝试新命令，每次都是不同的失败命令，前端的重复检测不触发，后端也没有任何干预手段。

### 问题 3：无跨请求轮次治理，理论上可无限循环

**现象**：任务恢复或前端重启后，轮次计数归零，agent 可重新获得完整循环预算，理论上可无限消耗资源。

**根因**：架构设计中轮次状态仅存在前端内存。前端的 `consecutiveMistakeCount` 是进程内变量，随任务实例生命周期存在，一旦任务 resume 或前端进程重启即归零。而后端作为所有请求的汇聚点，天然适合维护跨请求的持久状态，但原始版本未引入任何持久化存储（如 Redis）来追踪任务级轮次。这导致系统缺少一个不依赖前端生命周期的"硬上限"，即使前端计数失效，也没有后端兜底来终止失控的任务。

### 问题 4：并行工具调用与前端执行模型不匹配

**现象**：模型偶尔返回并行 tool_calls，前端处理时出现逻辑混乱。

**根因**：前后端对工具执行模式的约定不一致。前端基于 `didAlreadyUseTool` 机制设计——流中检测到第一个 tool_use 即中断，一轮只执行一个工具，执行完拿结果再进下一轮。这是 Roo Code 架构的固有设计，保证工具执行的顺序性和可追踪性。但后端原始版本未对请求参数做任何约束，`parallel_tool_calls` 走 LLM 默认值（通常为 true），模型可以返回多个并行 tool_calls。前端收到后只执行第一个并中断，其余工具调用被丢弃，导致模型认为已发出的工具未被响应，行为出现偏差。

## 三、针对性改进

### 改进 1：环境感知注入（EnvInjector）

后端新增 EnvInjector 处理器，自动检测项目类型（Django / pytest / Node / Go），向 system prompt 注入正确的测试运行器指导：

- **Django 项目**：注入 `runtests.py --settings=test_sqlite` 正确命令模板及错误命令禁令清单，从源头引导 agent 避免试错
- **非 Django 项目**：注入框架适配的轻量提示（如 pytest 配置检查、npm test、go test），避免无关噪声
- 知识来源为各框架官方贡献文档，非测试集泄露

### 改进 2：跨请求轮次治理 + 兜底终止

后端引入 Redis 跨请求轮次计数（按 TaskID，TTL 2 小时），即使任务恢复或前端重启，轮次仍然累计。达到上限时后端合成终止响应，优雅结束任务，提供最终兜底保障。

### 改进 3：重复命令检测

后端通过 Redis 追踪同一任务的命令执行历史，同一命令重复执行达阈值时触发告警，与前端已有的工具重复检测（连续相同调用 ≥3 次拦截）形成双重防护，阻断死循环。

### 改进 4：强制串行工具调用

后端强制设置 `parallel_tool_calls=false`，与前端单工具顺序执行模型对齐，消除并行 tool_calls 导致的处理混乱。

### 改进 5：生产场景适配

移除了评测专用的强制测试验证逻辑（`filterAttemptCompletion`），使生产环境中用户做简单修改时也能正常完成任务，同时保留对非生产场景的测试引导能力。基座从 v1.5.22 升级到 v1.6.2，获得最新上游修复。

## 四、SWE-bench 评测验证

使用 SWE-bench Lite 数据集，选取 10 个 Django 实例，模型 DeepSeek-V4-Pro，进行 5 轮对比评测。

### 核心指标对比

| 指标 | 改进前 | 改进后 | 改善 |

| 命令执行成功率 | 51.5% | 82.6% | **+31.1 个百分点** |
| 错误命令数 | 32 | 8 | **减少 75%** |
| 死循环实例 | 1/10 | 0/10 | **清零** |
| 平均工具调用数 | 21.3 | 18.4 | **减少 13.6%** |

### 指标定义

- **调用数**：agent 在该实例中发起的工具调用总次数（包括执行命令、读写文件、搜索等所有工具）。
- **错误命令（cmd_err）**：execute_command 工具产出的命令本身有误，未能成功执行。判定标准为命令输出中未出现测试运行摘要（如 `Ran N tests`），且包含以下任一错误标志：`Traceback`、`ModuleNotFoundError`、`ImportError`、`ImproperlyConfigured`、`AttributeError`、`SyntaxError`、`command not found`、`No such file`、`usage:` 等。这类错误意味着命令本身是错的（选错了测试运行器或调用方式），而非代码改动导致测试不通过。
- **命令成功率**：在所有 execute_command 调用中，命令本身正确并成功执行的比例。计算方式为 `(成功 + 测试失败) / 命令总数`。其中"成功"指命令无错误执行完毕，"测试失败"指命令正确执行、测试正常运行但未通过——后者说明命令本身没问题（测试运行器选对了），失败原因在于被测代码。因此两者均计入命令成功。只有"错误命令"（cmd_err）不计入成功。

### 关键结论

- **命令执行成功率大幅提升**：`python -m django test`（错误命令）从 12 次降至 0 次，`ImproperlyConfigured` 和 `ModuleNotFoundError` 错误完全消除。Agent 不再把轮次浪费在必败命令上。
- **死循环完全消除**：最严重的实例 django-11019 从 766 秒 / 45 次调用改善到 225 秒 / 25 次调用，全部实例无死循环。
- **解决率未退化**：改进聚焦于提升执行效率而非"解题"，解决率始终维持 80%，未解决的问题集合在 5 轮评测中完全一致，说明改进未引入副作用。
- **效率显著提升**：平均耗时降低三分之一，平均工具调用数下降，agent 行为更加高效。

---

## 五、总结

本次改进针对原始 agent loop 的四个核心问题（命令失败率高、死循环、无界轮次、并行工具混乱），实施了五项针对性改造。SWE-bench 评测验证表明，改进在保持解决率不变的前提下，命令执行成功率提升 31 个百分点，死循环清零，平均耗时降低 34%，有效提升了 agent 的执行效率和稳定性。

改进已构建为生产镜像 `zgsm/chat-rag:v1.6.2-agent-loop-fb`，并打包为离线部署包，具备上线条件。
