# Costrict 离线部署指导手册

> 目标：在断网 Ubuntu 机器上部署 Costrict 服务端，并接入本地大模型，最终用 `costrict` CLI 实现可对话的 TUI。
>
> 本手册基于对 `costrict-offline-deploy.tar.gz`（1.2G，含 14 个 Docker 镜像 + 配置 + 脚本）的逐文件分析编写，所有命令均可直接复制执行。

---

## 0. 你需要事先准备好的东西

在**有网机器**上准备好以下文件，再拷到断网机：

| # | 文件/资源 | 说明 |
|---|----------|------|
| 1 | `costrict-offline-deploy.tar.gz` | 本部署包（1.2G） |
| 2 | Docker + Compose 离线 deb 包 | 至少需要 `containerd.io`、`docker-ce`、`docker-ce-cli`、`docker-buildx-plugin`、`docker-compose-plugin` 五个 deb |
| 3 | 本地大模型推理服务 | OpenAI 兼容接口（vLLM / Ollama / TGI 等均可），需提供 `/v1/chat/completions`，最好也有 `/v1/completions`（代码补全用） |
| 4 | python3 + requests + pyyaml | 断网机自带或离线装好，`push-agent-rules.sh` 和 Nacos API 推送要用 |

> ⚠️ 第 3 项是整个系统能"对话"的前提。本部署包**不含任何模型权重**，只负责网关与业务服务。

## 1. 安装 Docker（断网机）

```bash
# 把 5 个 deb 放到同一目录，然后：
sudo dpkg -i *.deb
# 若有依赖缺失，按提示把依赖 deb 也补进目录重跑

# 验证
docker --version
docker compose version

# 让当前用户免 sudo 用 docker（需重新登录生效）
sudo usermod -aG docker $USER
```

---

## 2. 解压部署包

```bash
mkdir -p ~/costrict-server && cd ~/costrict-server
# 假设 tar.gz 已拷到本机
tar xzf /path/to/costrict-offline-deploy.tar.gz

# 解压后顶层结构（确认存在）：
#   docker-compose.yml   configure.sh   costrict.sh   load-images.sh
#   push-agent-rules.sh  verify-deploy.sh
#   images/              config/        scripts/
```

---

## 3. 修改 configure.sh（必须改）

`configure.sh` 是所有环境变量的定义文件，被模板渲染脚本和 APISIX 路由脚本 `source` 加载。用编辑器打开：

```bash
cd ~/costrict-server
nano configure.sh      # 或 vim configure.sh
```

**必须修改的项**（行号为参考）：

```bash
# --- 第 5 行：改成断网机本机 IP（不要写 127.0.0.1，要写内网/局域网 IP）---
COSTRICT_BACKEND="192.168.x.x"        # ← 改成本机实际 IP

# --- 第 14 行：本地大模型推理服务 IP（若与本机同机也写本机 IP）---
MODEL_SERVER_IP="192.168.x.x"         # ← 改成模型服务 IP

# --- 第 31-33 行：代码补全模型（默认是公网 dashscope，断网必须改）---
COMPLETION_BASEURL="http://192.168.x.x:8000/v1/completions"   # ← 本地推理服务的补全接口
COMPLETION_MODEL="你的补全模型名"       # 如 deepseek-coder-v2-lite
COMPLETION_APIKEY="你的key"             # 无 key 可填任意非空字符串
```

**其余项保持默认即可**（端口、数据库密码、OIDC client 等都已预置且互相配套）。密码建议跑通后再改。

---

## 4. 同步修改 docker-compose.yml（关键坑）

⚠️ `docker-compose.yml` 是**已渲染好的、写死的文件**（不是模板），`configure.sh` 改了不会自动同步。其中 `oidc-auth` 服务有两处写死了旧 IP，必须手动改：

```bash
cd ~/costrict-server
# 把 docker-compose.yml 里所有的 172.24.0.2 替换为你在 configure.sh 里设的本机 IP
sed -i 's/172.24.0.2/192.168.x.x/g' docker-compose.yml
```

验证已替换（应显示你设的 IP，不再有 172.24.0.2）：

```bash
grep -n "172.24.0.2" docker-compose.yml   # 应无输出
grep -n "SERVER_BASEURL\|PROVIDERS_CASDOOR_BASEURL" docker-compose.yml
```

## 5. 渲染配置模板

把 `.tpl` 模板里的 `{{变量}}` 用 `configure.sh` 的值渲染成实际配置：

```bash
cd ~/costrict-server
bash scripts/template_gen.sh
```

这会生成：`config/apisix/config.yaml`、`config/casdoor/app.conf`、`config/code-completion/config.yaml`、`config/quota-manager/app-conf.yml`、`config/prometheus/prometheus.yml` 等。

验证补全配置已渲染（应显示你设的补全地址，不再是 `{{...}}`）：

```bash
grep completionsUrl config/code-completion/config.yaml
```

## 8. 启动全部服务

```bash
cd ~/costrict-server
docker compose up -d
```

等待 30~60 秒，确认容器状态（核心服务都应为 Up）：

```bash
docker compose ps
```

预期看到这些容器为 Up：`apisix`、`etcd`、`redis`、`postgres`、`nacos`、`model-proxy`、`chat-rag`、`portal`、`casdoor`、`oidc-auth`、`code-completion`、`credit-manager`、`prometheus`、`grafana`。

若有容器不断重启，看日志定位：

```bash
docker compose logs --tail=50 <服务名>
```

## 9. 注册 APISIX 路由（必须做，否则服务不通）

> 说明：`costrict.sh install` 本应调用 `apisix_router_setting.sh` 来批量注册路由，但**该总控脚本不在部署包内**。这里手动执行 `scripts/apisix_router/` 下的所有脚本替代，效果相同。

```bash
cd ~/costrict-server
source ./configure.sh
cd scripts/apisix_router
for s in *.sh; do echo "== $s =="; sh "$s"; done
cd ../..
```

这些脚本通过 APISIX Admin API 注册 9 条路由，把外部路径转发到内部容器：

- `/chat-rag/api/v1/chat/*` → chat-rag（对话，带 OIDC 鉴权）
- `/code-completion/*` → code-completion（代码补全）
- `/ai-gateway/api/v1/models` → model-proxy（模型列表）
- `/oidc-auth/*`、`/credit/manager*`、`/quota-manager/*` → 各业务服务
- `/costrict/*` → portal（CLI 客户端二进制下载）
- `/login/oauth/*`、`/static/*`、`/casdoor/*` → casdoor（登录认证）

每个 curl 应返回 `HTTP/1.1 200` 或 `201`。若返回 401，检查 `configure.sh` 里 `APIKEY_APISIX_ADMIN` 与 `config/apisix/config.yaml` 是否一致（默认一致，勿乱改）。

## 10. 推送 agent_rules 到 Nacos

```bash
cd ~/costrict-server
bash push-agent-rules.sh ./config/chat-rag/agent_rules_nacos_v2.yaml
```

成功会显示 `agent_rules 推送成功`。这一步把对话 agent 的工作流规则写进 Nacos 配置中心（命名空间 `costrict`，group `CHAT_RAG`，dataId `agent_rules`），chat-rag 启动后从这里拉规则。

> 若提示获取 token 失败，说明 Nacos 还没就绪，等 30 秒重试。

## 11. 接入本地大模型（核心步骤）

Costrict 的对话模型走 **Nacos 配置中心**（不碰 `configure.sh`），调用链为：

```
CLI/客户端 ──► APISIX:39080 ──► chat-rag:8888 ──► model-proxy:8080 ──► 你的本地推理服务
                                                        ▲
                                              从 Nacos 拉 model-config
```

`model-proxy` 订阅 Nacos 的 `model-config`（命名空间 `costrict`，group `MODEL_PROXY`），从里面拿到模型真实地址再代理。

### 11.1 登录 Nacos 控制台

浏览器访问（也可在断网机本地用 curl 改）：

```
http://<本机IP>:31848/nacos
```

账号密码：`nacos` / `nacos`

### 11.2 修改 model-config

1. 左上角命名空间切换到 **`costrict`**
2. 配置管理 → 配置列表 → 筛选 group `MODEL_PROXY`
3. 找到 `dataId = model-config`，点击编辑
4. 修改每个模型项的 `privateConfig`，指向你的本地推理服务：

```yaml
models:
  - id: my-local-llm                      # 模型 id（自定义，会被 model_router 引用）
    publicInfo:
      maxTokens: 4096                     # 最大输出 token
      contextWindow: 32768                # 最大上下文窗口
      supportsImages: false               # 是否支持多模态图片
      description: 本地部署的对话模型
    privateConfig:
      convertedName: qwen2.5-72b-instruct # 真实模型名（推理服务认的名字）
      addr: http://192.168.x.x:8000/v1/chat/completions   # ← 你的本地推理服务地址
      extraHeaders:
        Authorization: Bearer sk-your-key # ← 你的 API key（无 key 填任意非空）
      skipSSLVerify: false
```

5. 点击「发布」。`model-proxy` 会自动订阅生效，无需重启。

### 11.3 修改 model_router（路由匹配）

同一命名空间下，找到 `dataId = model_router`（group `CHAT_RAG`），编辑：

```yaml
enabled: true
strategy: priority
priority:
  candidates:
    - modelName: "my-local-llm"           # ← 必须与上面 model-config 里的 id 一致
      enabled: true
      priority: 1
      weight: 5
  fallbackModelName: "my-local-llm"       # ← 也改成你的模型 id
  idleTimeoutMs: 180000
  totalIdleTimeoutMs: 360000
  maxRetryCount: 0
  retryIntervalMs: 5000
```

发布。chat-rag 收到对话请求后会按 `model_router` 选模型，转给 model-proxy，再转给你的本地推理服务。

### 11.4 验证模型链路

```bash
# 直接测 model-proxy 能否转发到你的推理服务
curl -s http://127.0.0.1:39080/ai-gateway/api/v1/models

# 测对话接口（需带 token，见下一步登录获取）
curl -s http://127.0.0.1:39080/chat-rag/api/v1/chat/completions \
  -H "Authorization: Bearer <你的token>" \
  -H "Content-Type: application/json" \
  -d '{"model":"my-local-llm","messages":[{"role":"user","content":"你好"}]}'
```

> 代码补全模型已在第 3 步通过 `configure.sh` + 模板渲染配置进 `config/code-completion/config.yaml`，无需进 Nacos。

## 14. 安装并启动 costrict CLI（实现对话 TUI）

### 14.1 取出 CLI 二进制

CLI 二进制已随包分发到 `data/portal/`，也可直接从源目录取：

```bash
# amd64 架构（绝大多数 x86 服务器）
mkdir -p ~/.costrict
cp config/portal/static_file/costrict/costrict/linux/amd64/1.1.34/costrict ~/.costrict/costrict
chmod +x ~/.costrict/costrict
# 建议加入 PATH
sudo ln -sf ~/.costrict/costrict /usr/local/bin/costrict

# arm64 架构改用：
# config/portal/static_file/costrict/costrict/linux/arm64/1.1.34/costrict
```

验证：

```bash
costrict version
# Version v1.1.34 ...
```

### 14.2 配置 BaseUrl 并登录

`costrict` CLI 需要知道后端地址（即 `configure.sh` 里的 `COSTRICT_BACKEND_BASEURL`）并完成 OAuth 登录获取 token。

```bash
# 设置后端地址（指向 APISIX 入口）
costrict auth        # 查看当前认证状态
```

初始化连接（按 CLI 引导配置 BaseUrl 为 `http://<本机IP>:39080`），然后通过 Casdoor 登录获取 Access Token。CLI 默认监听 `localhost:7999` 做本地服务管理。

> CLI 的连接配置参考 `costrict-config` 包内的 `costrict.json`（`listen: localhost:7999`，隧道通过 cotun 连后端）。本地部署时 cotun 隧道连的是 `data/portal` 提供的 portal 服务。

### 14.3 启动对话

```bash
costrict server      # 启动本地 http 管理服务
costrict check       # 检查与后端连通性
costrict service list # 查看可用服务
```

登录并连通后，即可在终端进入对话 TUI，效果类似当前会话——输入消息，模型流式返回。

> 若 `costrict check` 报连不上，依次确认：① 本机能 `curl http://<本机IP>:39080` 通；② APISIX 路由已注册（第 9 步）；③ 已在 Casdoor 登录拿到有效 token。

## 15. 常见问题排查

**Q: 某个容器一直重启**
```bash
docker compose logs --tail=80 <服务名>
```
最常见是 postgres 密码不一致（改了 `configure.sh` 密码但没改 `docker-compose.yml`）或 Nacos 连不上 postgres。

**Q: 对话返回 401/未授权**
APISIX 的 `/chat-rag/*` 路由开了 OIDC 鉴权，请求必须带有效 Casdoor token。确认 CLI 已登录，或检查 `oidc-auth` 容器日志。

**Q: 对话返回 502/模型错误**
model-proxy 转发失败。检查：① Nacos `model-config` 里的 `addr` 能否从 model-proxy 容器内访问（用 `docker exec model-proxy ping <模型IP>` 或 curl 测）；② `model_router` 的 `modelName` 与 `model-config` 的 `id` 是否一致。

**Q: 代码补全不工作**
确认第 3 步改了 `COMPLETION_*` 并重跑 `template_gen.sh`，然后重启容器：`docker compose restart code-completion`。本地推理服务需提供 `/v1/completions`（FIM 补全）接口。

**Q: `costrict.sh install` 报 `apisix_router_setting.sh` 不存在**
这是部署包的已知缺失。直接按本手册第 5~10 步手动执行即可，不要用 `costrict.sh install`。

**Q: Nacos 控制台看不到 `costrict` 命名空间下的配置**
确认命名空间已切换到 `costrict`（左上角下拉）。预置配置都在该命名空间下，group 为 `CHAT_RAG` 或 `MODEL_PROXY`。
